Foundation
Naive RAG
The classic retrieve-then-generate pipeline
The simplest form of RAG. Chunk your documents, embed them, store in a vector DB, retrieve top-K chunks on query, and pass to the LLM. No frills, but it works surprisingly well for many cases.
▼ Show details, use cases & code
📌 Use Cases
📚Internal knowledge base Q&A (HR docs, policies)
🛠️Simple product documentation chatbots
🎓Course material assistants for students
📋FAQ automation for customer support
⚖️ Pros & Cons
✓ Easy to implement
✗ Retrieval quality bottleneck
✓ Low latency
✗ No iterative reasoning
✓ Low cost
✗ Fails on complex queries
💻 Pipeline Pseudocode
# Naive RAG Pipeline chunks = split_documents(docs) vectors = embed(chunks) store(vectors, vector_db) query_vec = embed(user_query) top_k = similarity_search(query_vec, k=5) answer = llm(user_query + top_k)

Foundation
Advanced RAG
Pre & post-retrieval optimization
Enhances Naive RAG with query rewriting before retrieval, better chunking strategies (semantic/sliding window), and post-retrieval re-ranking to push the best chunks to the top before passing to the LLM.
▼ Show details, use cases & code
📌 Use Cases
⚖️Legal research assistants requiring precise document retrieval
🏥Medical literature search with clinical accuracy needs
💼Enterprise search over large heterogeneous document sets
🔬Scientific paper Q&A needing methodology-level answers
⚖️ Pros & Cons
✓ Better retrieval precision
✗ More engineering complexity
✓ Re-ranking improves quality
✗ Slightly higher latency
✓ Handles ambiguous queries
✗ More components to tune
💻 Pipeline Pseudocode
# Advanced RAG better_query = rewrite_query(user_query) candidates = retrieve(better_query, k=20) top_k = rerank(candidates, user_query, k=5) answer = llm(user_query + top_k)

Foundation
Modular RAG
Plug-and-play component architecture
Treats each stage as a swappable module — retriever, reranker, memory, reader, fusion — enabling custom pipelines. You can mix dense + sparse retrieval, add web search, plug in any LLM, and compose workflows.
▼ Show details, use cases & code
📌 Use Cases
🏢Production RAG systems needing A/B testing of components
🔀Multi-source pipelines (docs + web + database)
🚀RAG platforms serving multiple teams with different needs
🧪Research environments comparing retrieval strategies
⚖️ Pros & Cons
✓ Maximum flexibility
✗ High upfront design cost
✓ Easy to swap components
✗ Complex orchestration
✓ Mix any retrievers
✗ Integration testing burden
💻 Architecture Pattern
# Modular RAG pipeline = Pipeline([ QueryRewriter(), # module 1 HybridRetriever(), # module 2 CrossEncoderReranker(), # module 3 ContextCompressor(), # module 4 OpenAIReader() # module 5 ]) answer = pipeline.run(user_query)

Index Type
Dense RAG
Semantic vector similarity search
Uses dense embedding models (like text-embedding-3, BGE, E5) to encode documents and queries into high-dimensional vectors. Retrieval is semantic — finds meaning-similar content even without exact keyword match.
▼ Show details, use cases & code
📌 Use Cases
💬Conversational search where synonyms matter
🌍Multilingual document retrieval
📰News article similarity and clustering
🛒Product recommendation from description matching
⚖️ Pros & Cons
✓ Semantic understanding
✗ Misses exact keyword matches
✓ Cross-lingual capable
✗ Embedding model quality matters
✓ Handles paraphrasing
✗ ANN index build time
💻 Implementation
# Dense Retrieval model = SentenceTransformer("BAAI/bge-large") doc_vecs = model.encode(chunks) index = faiss.IndexFlatIP(dim) index.add(doc_vecs) q_vec = model.encode([query]) scores, ids = index.search(q_vec, k=5)
🔑
Index Type
Sparse RAG
Keyword-based BM25 / TF-IDF retrieval
Uses traditional IR methods like BM25 that score documents based on exact keyword frequency and inverse document frequency. Extremely reliable when exact terminology matters — IDs, product names, error codes, technical terms.
▼ Show details, use cases & code
📌 Use Cases
🆔Searching by product SKU, invoice number, error codes
⚕️Medical records with ICD-10 codes and drug names
⚖️Legal case search by statute number or citation
💻Code search by function name or library reference
⚖️ Pros & Cons
✓ Exact match reliability
✗ No semantic understanding
✓ No GPU needed
✗ Fails on paraphrasing
✓ Fast, battle-tested
✗ Keyword mismatch = miss
💻 Implementation
# BM25 Sparse Retrieval tokenized = [doc.split() for doc in chunks] bm25 = BM25Okapi(tokenized) scores = bm25.get_scores(query.split()) top_k = argsort(scores)[:-6:-1]
🔀
Index Type
Hybrid RAG
Dense + Sparse fusion (best of both)
Combines dense semantic search with sparse keyword search. Results are merged using Reciprocal Rank Fusion (RRF) or weighted scoring. Catches both semantic matches and exact keyword hits — the production-ready default.
▼ Show details, use cases & code
📌 Use Cases
🏦Financial research mixing terminology and meaning
🛒E-commerce search (exact SKU + semantic description)
🏥Healthcare: ICD codes + clinical text understanding
🔧DevOps log search mixing error codes + natural language
⚖️ Pros & Cons
✓ Best overall recall
✗ Higher infra cost
✓ Handles mixed queries
✗ Fusion weight tuning needed
✓ Production proven
✗ 2 indexes to maintain
💻 RRF Fusion
# Reciprocal Rank Fusion dense_results = vector_db.search(query, k=20) sparse_results = bm25.search(query, k=20) for rank, doc in results: score += 1 / (k + rank) # RRF formula top_k = sort_by_rrf_score(merged)

Index Type
Graph RAG
Knowledge graph + entity relationship retrieval
Builds a knowledge graph from documents — extracting entities (people, orgs, concepts) and relationships between them. Retrieval traverses the graph, enabling answers that require connecting facts across multiple documents.
▼ Show details, use cases & code
📌 Use Cases
🔍Investigative research: "How are Person A and Company B connected?"
💊Drug-interaction mapping across medical literature
🏢Org chart / corporate hierarchy reasoning
🧬Gene-protein-disease relationship queries in biomedical research
⚖️ Pros & Cons
✓ Multi-hop reasoning
✗ Expensive graph construction
✓ Captures relationships
✗ Entity extraction errors propagate
✓ Explainable reasoning paths
✗ Complex to maintain
💻 Graph Construction
# Graph RAG (e.g. Microsoft GraphRAG) entities, relations = extract_kg(docs) graph = Neo4j() graph.add(entities, relations) subgraph = graph.traverse( start=entity_from(query), depth=3 ) answer = llm(query + subgraph)

Query Strategy
Multi-Hop RAG
Chain of retrieval steps for complex queries
Each retrieval step informs the next. The model retrieves initial context, reasons about what's still missing, retrieves again based on that reasoning, and repeats — like following a chain of clues to reach the final answer.
▼ Show details, use cases & code
📌 Use Cases
📜"Who founded the company that acquired X?" — requires chaining facts
🏛️Legal reasoning: statute → case law → precedent
🔭Scientific hypothesis exploration across papers
🕵️Investigative journalism connecting events across time
⚖️ Pros & Cons
✓ Handles bridge questions
✗ Latency multiplies with hops
✓ Reasons across documents
✗ Error can compound per hop
✓ Structured reasoning
✗ Hard to determine when to stop
💻 Multi-Hop Loop
# Multi-Hop Retrieval context = [] current_query = user_query for hop in range(max_hops): docs = retrieve(current_query) context.append(docs) current_query = llm_followup( user_query, context ) if has_answer(context): break answer = llm(user_query + context)
🔄
Query Strategy
Iterative RAG
Retrieve → reason → retrieve loop
The LLM iteratively retrieves more information until it has enough to answer confidently. Unlike multi-hop (fixed structure), iterative RAG lets the model decide when to keep searching — more dynamic and flexible.
▼ Show details, use cases & code
📌 Use Cases
📊Complex business analysis needing multiple data points
🤔Open-ended research where depth is unknown upfront
📝Report generation from large document collections
🩺Differential diagnosis requiring iterative symptom lookup
⚖️ Pros & Cons
✓ Adaptive depth
✗ Unpredictable latency
✓ Self-directed research
✗ Risk of infinite loops
✓ Natural stopping logic
✗ More LLM calls = higher cost
💻 Iteration Pattern
# Iterative RAG context, iterations = [], 0 while not sufficient(context): next_q = llm_plan(user_query, context) new_docs = retrieve(next_q) context += new_docs iterations += 1 if iterations > max_iter: break final = llm(user_query + context)
🎯
Query Strategy
Adaptive RAG
Decides whether to retrieve at all
Classifies queries by complexity and routes them: simple factual questions go straight to the LLM (no retrieval needed), moderate queries use single-step RAG, complex ones use multi-step RAG. Saves cost and latency.
▼ Show details, use cases & code
📌 Use Cases
💬General-purpose chatbots handling both simple and deep questions
⚡High-volume apps where retrieval cost must be minimized
🤖Customer service bots with mixed query complexity
📱Mobile AI assistants with strict latency budgets
⚖️ Pros & Cons
✓ Cost-efficient
✗ Router must be accurate
✓ Optimal latency per query
✗ Misclassification causes errors
✓ Scales well
✗ More routing logic to maintain
💻 Query Router
# Adaptive RAG Router complexity = classify_query(user_query) # → "simple" | "single_hop" | "multi_hop" if complexity == "simple": answer = llm_direct(user_query) elif complexity == "single_hop": answer = naive_rag(user_query) else: answer = multi_hop_rag(user_query)

Generation
RAG Fusion
Multi-query generation + result merging
Generates 3–5 variations of the user's query, retrieves documents for each, then merges all result sets using Reciprocal Rank Fusion. This expands coverage and reduces the chance of missing relevant content due to query phrasing.
▼ Show details, use cases & code
📌 Use Cases
🔍Research assistants where query phrasing varies widely
🌐Multi-language corpora requiring perspective-diverse retrieval
📖Academic literature review needing comprehensive coverage
🏷️Product search where users describe items differently
⚖️ Pros & Cons
✓ Higher recall coverage
✗ 3-5x retrieval cost
✓ Diverse perspectives
✗ More LLM calls for query gen
✓ Less query sensitivity
✗ Latency increase
💻 Fusion Pipeline
# RAG Fusion sub_queries = llm_generate_queries( user_query, n=4 ) all_results = [] for q in sub_queries: all_results.append(retrieve(q, k=10)) merged = reciprocal_rank_fusion(all_results) answer = llm(user_query + merged[:5])

Generation
Self-RAG
LLM decides when to retrieve & self-critiques
The LLM is fine-tuned to generate special reflection tokens — [Retrieve], [Relevant], [IsSupported], [IsUseful] — to decide when to search, whether retrieved docs are relevant, and whether the generated answer is grounded.
▼ Show details, use cases & code
📌 Use Cases
🩺Medical Q&A needing grounded, verifiable answers
⚖️Legal advice systems requiring citation grounding
📰Fact-checking assistants that need to verify claims
🎓Educational platforms where accuracy is non-negotiable
⚖️ Pros & Cons
✓ Reduces hallucination
✗ Requires fine-tuned model
✓ Self-verifying outputs
✗ Complex training pipeline
✓ Smart retrieval decisions
✗ Not plug-and-play
💻 Reflection Tokens
# Self-RAG (simplified) output = model.generate(query) # Model outputs e.g.: # "[Retrieve=Yes] The answer..." # "[IsSupported=Fully]" # "[IsUseful=5]" if output.needs_retrieval(): docs = retrieve(query) output = model.generate(query + docs) return output.best_passage()
✏️
Generation
Corrective RAG (CRAG)
Evaluates retrieval quality, falls back to web
After retrieval, a lightweight evaluator scores document relevance. If score is high → proceed. If ambiguous → refine. If low → trigger a web search to supplement or replace the retrieved content entirely. Graceful degradation.
▼ Show details, use cases & code
📌 Use Cases
📰News/current events where static KB may be outdated
🌐General-purpose assistants with partial knowledge bases
🔬Research tools where no-answer is worse than web search
💼Enterprise systems with incomplete internal docs
⚖️ Pros & Cons
✓ Graceful fallback
✗ Evaluator must be accurate
✓ Avoids bad context
✗ Web search adds latency
✓ High answer quality
✗ Two retrieval systems needed
💻 Corrective Loop
# CRAG Pipeline docs = retrieve(query) score = evaluate_relevance(docs, query) if score > 0.8: context = docs # ✓ Good elif score > 0.4: context = refine(docs) # ~ Ambiguous else: context = web_search(query) # ✗ Bad answer = llm(query + context)
💡
Query Strategy
HyDE
Hypothetical Document Embeddings
Instead of embedding the query directly, the LLM generates a hypothetical document that would answer the query, then embeds that. This closes the gap between short queries and long document chunks in the vector space.
▼ Show details, use cases & code
📌 Use Cases
🔬Scientific search where queries are very short but docs are long
📚Library/archive search with sparse queries
🩺Clinical notes retrieval from brief diagnostic codes
🧑‍💻Code search from natural language function descriptions
⚖️ Pros & Cons
✓ Better query-doc alignment
✗ Hypothetical doc may hallucinate
✓ Improves recall on short queries
✗ Extra LLM call per query
✓ Simple to implement
✗ Quality depends on LLM
💻 HyDE Pipeline
# Hypothetical Document Embeddings hypo_doc = llm( f"Write a passage that answers: {query}" ) # Use hypo doc for retrieval, not query vec = embed(hypo_doc) top_k = vector_db.search(vec, k=5) answer = llm(query + top_k)
🤖
Advanced
Agentic RAG
LLM agent orchestrates retrieval with tools
An LLM agent uses retrieval as one of many tools — alongside web search, calculators, code execution, APIs. It plans multi-step, decides which tool to use, and iterates until the task is solved. The future of RAG.
▼ Show details, use cases & code
📌 Use Cases
📊Financial analysis: search docs + pull live data + run calculations
🔧DevOps assistant: search runbooks + check monitoring APIs + take action
🏥Clinical decision support: search literature + EHR data + guidelines
🛒Autonomous shopping agent: search catalog + compare + place orders
⚖️ Pros & Cons
✓ Most capable pattern
✗ Highest cost & latency
✓ Handles any task complexity
✗ Harder to control/audit
✓ Self-correcting
✗ Needs careful safety design
💻 Agent Loop
# Agentic RAG tools = [vector_search, web_search, sql_query, calculator, code_exec] agent = ReActAgent(llm, tools) result = agent.run(task) # Agent internally: # Thought → Action → Observation # → Thought → Action → ... # → Final Answer
⚡
Advanced
Speculative RAG
Small model drafts, large model verifies
A smaller, faster LLM generates a draft answer speculatively using retrieved context. Then a larger, more capable LLM verifies and refines it. Reduces latency and cost while maintaining quality — like speculative decoding for RAG.
▼ Show details, use cases & code
📌 Use Cases
⚡High-throughput APIs needing fast responses at scale
💰Cost-constrained systems wanting GPT-4 quality at lower cost
📞Real-time voice assistants with strict latency SLAs
🏭Production systems with mixed simple/complex query loads
⚖️ Pros & Cons
✓ Lower cost than full large LLM
✗ Two models to manage
✓ Faster time-to-first-token
✗ Verification overhead
✓ High quality output
✗ Complex orchestration
💻 Draft-Verify Pattern
# Speculative RAG docs = retrieve(query) # Fast small model drafts draft = small_llm.generate(query + docs) # Large model verifies/refines final = large_llm.verify( query, docs, draft )

Index Type
Structured RAG
NL-to-SQL over databases and tables
Instead of unstructured text, retrieval is from SQL databases, spreadsheets, or APIs. The LLM converts the natural language query into SQL (Text-to-SQL), executes it, and uses the structured results as context to generate the answer.
▼ Show details, use cases & code
📌 Use Cases
📊Business intelligence — "What were last quarter's top 5 products?"
🏦Banking dashboards querying transaction databases
🏥Hospital analytics over EHR databases
🛒E-commerce inventory and sales reporting via NL
⚖️ Pros & Cons
✓ Precise, structured answers
✗ SQL generation can fail on complex schemas
✓ Works on live data
✗ Needs schema context in prompt
✓ No embedding needed
✗ DB access security risk
💻 Text-to-SQL Pattern
# Structured RAG sql = llm( schema + f"Convert to SQL: {query}" ) rows = db.execute(sql) answer = llm( f"Answer {query} given: {rows}" )

Multimodal RAG
Retrieve images, tables, charts, audio
Extends RAG beyond text to retrieve and reason over images, charts, diagrams, PDFs with mixed content, audio transcripts, or video frames. Uses multimodal embeddings (CLIP, ImageBind) and multimodal LLMs (GPT-4V, Gemini).

🩻Medical imaging: retrieve similar X-rays + clinical notes
📐Architecture/engineering: search technical drawings + specs
🎥Video understanding: retrieve relevant scenes from footage
⚖️ Pros & Cons
✓ Full document understanding
✗ Complex multimodal indexing
✓ Works on charts/diagrams
✗ Higher compute cost
✓ Rich context retrieval
✗ Fewer production-ready tools

🧭 When to Use Which RAG
📁 What's your data source?
Unstructured text docs → Dense / Hybrid
SQL / spreadsheets → Structured RAG
Images, PDFs, charts → Multimodal RAG
Knowledge graph → Graph RAG
❓ How complex are queries?
Simple factual Q&A → Naive RAG
Need multiple facts chained → Multi-Hop RAG
Mixed simple + complex → Adaptive RAG
Open-ended research → Iterative / Agentic
⚡ What's your priority?
Speed + low cost → Naive / Adaptive RAG
Maximum accuracy → Advanced / Self-RAG
High recall coverage → RAG Fusion / Hybrid
Incomplete KB fallback → CRAG
🔍 What retrieval issue do you have?
Missing exact keywords → Sparse or Hybrid
Query-doc length mismatch → HyDE
Query too vague → RAG Fusion
Relationships between docs → Graph RAG
